# GoFront v0.0.4 — Roadmap & Spec Gap Analysis

## Methodology

Systematic audit of [The Go Programming Language Specification (go1.26)](https://go.dev/ref/spec)
against GoFront's current feature set. Each gap is evaluated against the
**no-runtime** philosophy: features must compile to clean JS with at most tree-shaken
helpers. Features requiring a scheduler, runtime type metadata, or raw memory access are
excluded.

---

## Tier 1 — High value, pure compile-time (deferred to v0.0.5)

### 1. Method expressions (`T.Method`)

**Go spec §Method expressions.** `T.Method` yields a plain function whose first parameter
is the receiver. `(*T).Method` works similarly for pointer receivers.

```go
type Point struct { X, Y int }
func (p Point) Dist() float64 { ... }

f := Point.Dist    // func(Point) float64
f(Point{3, 4})     // equivalent to Point{3, 4}.Dist()
```

**Why it fits:** Pure compile-time transform — wrap the method in a closure that takes the
receiver explicitly. Zero runtime cost.

**Effort:** Medium. Parser already resolves `SelectorExpr`; the type-checker needs to
detect `TypeName.MethodName` (instead of value.field) and produce a func type with the
receiver prepended to params. Codegen emits a lambda.

---

### 2. Method values (bound method references)

**Go spec §Method values.** `x.Method` stored in a variable must bind the receiver.

```go
p := Point{3, 4}
f := p.Dist        // f captures p as receiver
f()                // calls p.Dist()
```

**Current bug:** Codegen emits `p.Dist` without `.bind(p)`, so the receiver is lost when
the method is called later.

**Why it fits:** Codegen adds `.bind(obj)` when a method selector is used as a value (not
immediately called). No runtime helper needed.

**Effort:** Low–Medium. Detect "method used as value" (selector not followed by call) in
codegen and emit `.bind()`.

---

### 3. Struct and array equality

**Go spec §Comparison operators.** Structs and arrays are comparable with `==`/`!=` if all
fields (or elements) are comparable. Go performs deep value comparison.

```go
type Point struct { X, Y int }
a := Point{1, 2}
b := Point{1, 2}
a == b  // true in Go, false in GoFront (reference comparison)
```

**Current behavior:** `==` compiles to `===`, which is reference equality in JS. Two
distinct struct instances with identical fields compare as `false`.

**Why it fits:** Emit a tree-shaken `__equal(a, b)` helper that does field-by-field
comparison. The type-checker already knows the struct layout. Alternatively, inline the
comparison for small structs.

**Effort:** Medium. Need to:
- Type-checker: verify both operands are comparable (reject slices, maps, functions).
- Codegen: replace `===` with `__equal(a, b)` for struct/array operands.
- Emit `__equal` helper (tree-shaken).
- Handle nested structs recursively, embedded fields.

---

### 4. Terminating statement analysis

**Go spec §Terminating statements.** The compiler must verify that functions with return
types have a terminating statement on every code path.

```go
func f(x int) int {
    if x > 0 {
        return x
    }
    // Go compiler error: missing return
}
```

**Current behavior:** GoFront silently accepts this — no missing-return check.

**Why it fits:** Pure type-checker analysis, no runtime cost. Improves error reporting.

**Effort:** Medium. Implement `isTerminating(stmt)` per the Go spec rules (return, panic,
if-with-else both terminating, for-without-break, switch-with-default-all-terminating, etc.)
and check every non-void function body's last statement.

---

### 5. Const expression repetition in `const` blocks

**Go spec §Constant declarations, §Iota.** In a parenthesized `const` block, omitting the
expression repeats the previous expression (with incremented `iota`).

```go
const (
    A = iota * 2    // 0
    B               // 2  (repeats iota * 2)
    C               // 4
)
const (
    Read   = 1 << iota  // 1
    Write               // 2
    Exec                // 4
)
```

**Current behavior:** Only plain `iota` substitution works. `const ( A = iota * 2; B; C )`
gives B=1, C=2 instead of B=2, C=4 because the parser substitutes a literal integer rather
than repeating the full expression with `iota` replaced.

**Why it fits:** Pure parse/check-time. The parser needs to clone the previous `ConstSpec`'s
expression AST and substitute `iota` with the current index, instead of just inserting a
bare integer literal.

**Effort:** Low–Medium. Implement AST cloning with iota substitution in the parser's const
block handler.

---

### 6. Exported/unexported identifier enforcement

**Go spec §Exported identifiers.** Identifiers starting with an uppercase letter are
exported; lowercase identifiers are unexported and must not be accessed from other packages.

**Current behavior:** Cross-package lowercase access is allowed (noted in semantic
differences table). This is a correctness gap.

**Why it fits:** Pure type-checker check during cross-package name resolution. Zero runtime
cost.

**Effort:** Low. In the type-checker's cross-package member resolution, check that the
identifier's first character is uppercase; emit an error otherwise.

## Tier 2 — Moderate value, small effort

### 7. String indexing returns byte

**Go spec §Index expressions.** `s[i]` on a string returns a `byte` (uint8), not a
single-character string.

**Current behavior:** `s[i]` compiles to `s[i]` in JS, which returns a single-character
string. Go programs that do arithmetic on `s[i]` (like `s[i] - '0'`) would fail.

**Fix:** Compile `s[i]` to `s.charCodeAt(i)` when the type-checker knows `s` is a string.
The `charCodeAt` result is a number, matching Go's byte semantics.

**Effort:** Low. Detect string indexing in codegen and emit `.charCodeAt()`.

---

### 8. Additional stdlib shims

Useful standard library packages that map cleanly to JS built-ins:

| Package | Functions | JS mapping |
|---------|-----------|------------|
| `bytes` | `Contains`, `HasPrefix`, `HasSuffix`, `Join`, `Split`, `Equal`, `Compare`, `Index`, `Repeat`, `ToUpper`, `ToLower`, `TrimSpace`, `NewReader`, `Buffer` | Similar to `strings` shim but operating on `Uint8Array` |
| `unicode` | `IsLetter`, `IsDigit`, `IsSpace`, `IsUpper`, `IsLower`, `ToUpper`, `ToLower` | Regex or char-code checks |
| `os` | `Exit`, `Args`, `Getenv` | `process.exit`, `process.argv`, `process.env` |
| `path` | `Join`, `Base`, `Dir`, `Ext`, `Clean` | String manipulation |
| `fmt` | `Sscanf`, `Fscanf` | Regex-based parsing |

**Why it fits:** Follow existing pattern — namespace types that inline to JS calls. No
runtime helpers beyond what's already tree-shaken.

**Effort:** Low–Medium per package. `unicode` and `os` are easiest. `bytes` is medium
(parallel to `strings` shim). `path` is medium.

---

### 9. Conversion from slice to array (`[N]T(slice)`)

**Go spec §Conversions (Go 1.20).** Converting a slice to a fixed-size array type.

```go
s := []int{1, 2, 3}
a := [3]int(s)       // a is [3]int{1, 2, 3}
a4 := [4]int(s)      // runtime panic: len < 4
```

**Why it fits:** Compile-time type annotation + optional runtime bounds check. With the
v0.0.5 array semantics plan in place, this becomes a natural extension.

**Effort:** Low (after v0.0.5 array work). Parser already handles type conversions; add
array-type conversion handling.

---

### 10. Multi-value function forwarding

**Go spec §Calls.** If `g()` returns `(A, B)` and `f` takes `(A, B)`, then `f(g())` is
valid and forwards the return values.

```go
func split(s string) (string, string) { ... }
func join(a, b string) string { ... }
join(split("hello world"))
```

**Why it fits:** Compile-time transform. Codegen detects the pattern and emits
`join(...split("hello world"))` or uses a temp variable.

**Effort:** Medium. Need type-checker support to recognize the multi-value-forwarding
pattern and codegen to emit the spread or temp.

## Tier 1b — Confirmed bugs (high priority fixes)

### 11. Blank identifier `_ = expr` in assignments

**Go spec §Blank identifier.** `_` may appear on the left side of any assignment to discard
a value. This is extremely common in Go for ignoring return values.

```go
_ = someFunc()       // discard return value
_, err := os.Open(f) // discard first return value
```

**Current bug:** `_ = expr` fails with `Undefined: '_'`. The type-checker calls
`scope.lookup("_")` on the LHS of an `AssignStmt`, which returns null → error. The blank
identifier is only handled in `DefineStmt` (`:=`) and `for...range`, NOT in regular `=`
assignments.

**Effort:** Low. Skip `_` in AssignStmt LHS checking; suppress output in codegen.

---

### 12. Positional (unkeyed) struct literals

**Go spec §Composite literals — Struct literals.** Struct fields may be initialized
positionally without keys: `Point{1, 2}` is equivalent to `Point{X: 1, Y: 2}`.

```go
type Point struct { X, Y int }
p := Point{1, 2}   // X=1, Y=2 by field order
```

**Current bug:** `Point{1, 2}` generates `new Point({ })` — an empty struct! The codegen's
`_genStructFields` filters to `KeyValueExpr` elements only, discarding positional args.

**Also broken:** Type-elided positional structs in slices: `[]Point{{1, 2}}` generates
`[[1, 2]]` instead of `[new Point({X: 1, Y: 2})]`.

**Effort:** Medium. Detect non-keyed composite lit elements and map them to fields by
declaration order from the struct type.

---

### 13. Assigning values to `interface{}` variables

**Go spec §Assignability.** Any type satisfies the empty interface `interface{}`, so
`var x interface{} = 42` and `x = someStruct` must be accepted.

```go
var x interface{} = 42   // valid Go — rejected by GoFront
func show(x interface{}) {}
show(42)                 // valid Go — rejected by GoFront
```

**Current bug:** The type-checker rejects assigning any concrete value to an `interface{}`
typed variable or parameter. The `any` alias works correctly (`var x any = 42` is accepted),
but writing the equivalent `interface{}` explicitly fails with `"int does not implement
interface{...}"`. The same bug prevents assigning *typed* values like `var n int = 42;
var x interface{} = n`.

**Note:** `any` works. `interface{}` doesn't. Passing concrete types to `func(any)` works;
passing them to `func(interface{})` doesn't. This is a regression in the assignability
check.

**Effort:** Low–Medium. The `assertAssignable` function needs to accept any concrete type
when the target is an empty interface (interface with zero methods), whether it's spelled
`any` or `interface{}`. Also ensure *typed* values can be assigned to `any` variables via
`=` (not just `:=`).

---

### 14. Comma-ok patterns with `=` (not just `:=`)

**Go spec §Index expressions, §Type assertions.** The comma-ok pattern works with both `:=`
and `=` assignment.

```go
var v int
var ok bool
v, ok = m["key"]     // map comma-ok with =
v, ok = x.(int)      // type assertion comma-ok with =
```

**Current bug:** Both patterns only work with `:=`. With `=`:
- Map: `v, ok = m["key"]` generates `[v, ok] = (m["a"] ?? 0)` — destructuring a scalar.
- Type assert: The `_commaOk` flag is only set for `DefineStmt`, not `AssignStmt`.

**Effort:** Low. Mirror the existing `DefineStmt` comma-ok detection logic in `AssignStmt`
for both typechecker and codegen.

---

### 15. `for range` string yields runes (integers), not JS characters

**Go spec §For range.** Iterating over a string produces `(index int, value rune)` where
`rune` is an integer (Unicode code point).

```go
for _, c := range "AB" {
    println(c == 65)    // true in Go (c is rune 'A' = 65)
    println(c + 1)      // 66 in Go
}
```

**Current bug:** The range variable `c` is a JS string character, not an integer.
`c == 65` is `false` (string vs number), and `c + 1` produces `"A1"` (string
concatenation).

**Why it fits:** Change codegen to use `.codePointAt()` for the value variable in string
range loops. No runtime helper needed.

**Effort:** Low–Medium. Codegen needs to wrap the string range value extraction with
`.codePointAt()`. The type-checker already types the second range variable as `rune`/`int`,
so inference is correct — it's purely a codegen fix.

---

### 16. Type switch with capture variable (`v := x.(type)`)

**Go spec §Type switches.** `switch v := x.(type) { case T: use(v) }` must declare `v` in
each case clause, typed according to the case type.

```go
switch v := x.(type) {
case int:
    println(v + 1)    // v is int here
case string:
    println(v)        // v is string here
}
```

**Current bug:** When `v` is used in multiple case branches, the type-checker reports
`'v' declared and not used` for one or more cases. A single-case type switch works;
multi-case fails with the unused-variable error.

**Effort:** Low–Medium. The type-checker's scope handling for `v` needs to treat each case
clause as having its own declaration of `v` with the narrowed type, and mark it used
independently per clause.

## Tier 3 — Lower priority / more effort

### 17. Complete rune literal escape sequences (renumbered)

**Go spec §Rune literals.** Go supports `\u` (4 hex), `\U` (8 hex), `\x` (2 hex), and
octal (`\000`) escapes in rune and string literals.

**Current status:** Basic escapes (`\n`, `\t`, `\\`, `\'`, `\"`) are handled. Need to
verify `\u`, `\U`, `\x`, and octal three-digit escapes are fully supported.

**Effort:** Low if already handled; Medium if not.

---

### 18. Improved numeric type conversions

**Go spec §Conversions between numeric types.** Converting `float64` to `int` should
truncate toward zero. Converting between sized integer types should behave correctly
(sign-extend or truncate).

**Current behavior:** `int(x)` compiles to `Math.trunc(x)` or similar. But does
`int8(300)` correctly produce 44 (truncation to 8-bit signed)?

**Why it fits:** Compile-time transform — emit `(x << 24) >> 24` for int8 truncation, etc.
No runtime helper needed for basic cases.

**Effort:** Medium. Need to map each sized type conversion to the right bit-manipulation
expression.

---

### 19. `panic` with `recover` improvements

**Go spec §Handling panics.** `recover()` should capture the panic value and allow the
goroutine to continue. Current `defer` + `recover` likely works for basic cases but edge
cases may be missing.

**Effort:** Low — verify current try/catch/finally implementation covers main patterns.

---

### 20. `for` range over map with delete

**Go spec §For range.** Deleting map entries during range iteration should not crash and
the deleted key must not appear later. JS `for...in` on objects handles this correctly, but
should be verified.

**Effort:** Low — mostly testing/verification.

---

## Explicitly out of scope

| Feature | Reason |
|---------|--------|
| Goroutines / channels / `select` | Requires scheduler/runtime |
| `unsafe` package | Requires memory model |
| `reflect` package | Requires runtime type metadata |
| `cgo` | FFI outside JS scope |
| `goto` | No clean JS translation |
| Exact integer overflow/wrapping | JS numbers are float64 |
| Random map iteration order | JS objects are insertion-ordered |
| Full UTF-8 string byte semantics | JS strings are UTF-16; matching Go byte semantics everywhere would require pervasive `TextEncoder/Decoder` calls |
| Generic type aliases (Go 1.24) | Requires generics first (v0.0.5); may revisit in v0.0.6 |
| `comparable` constraint (Go 1.18+) | Requires generics first (v0.0.5) |

## Proposed v0.0.4 scope

**Theme: Spec compliance & bug fixes** — close the most impactful correctness gaps with
low-effort changes. Medium-effort features are deferred to v0.0.5.

| # | Feature | Tier | Effort | Impact |
|---|---------|------|--------|--------|
| 1 | Method expressions (`T.Method`) | 1 | Medium | Unlocks functional Go patterns |
| 2 | Method values (`.bind()`) | 1 | Low–Med | Fixes a correctness bug |
| 3 | Struct/array equality | 1 | Medium | Widely expected by Go developers |
| 4 | Terminating statement analysis | 1 | Medium | Catches real bugs at compile time |
| 5 | Const expression repetition | 1 | Low–Med | Enables idiomatic `iota` patterns |
| 6 | Exported/unexported enforcement | 1 | Low | Package encapsulation correctness |
| 7 | String indexing → `.charCodeAt()` | 2 | Low | Fixes silent string-as-byte bugs |
| 8 | Stdlib shims (`unicode`, `os`) | 2 | Low–Med | Broader Go program compatibility |
| 9 | Slice-to-array conversion | 2 | Low | Depends on v0.0.5 array work |
| 10 | Multi-value function forwarding | 2 | Medium | Common Go idiom |
| 11 | Blank identifier `_ = expr` | 1b | Low | **Bug** — very common Go idiom, currently rejected |
| 12 | Positional struct literals (`P{1, 2}`) | 1b | Medium | **Bug** — generates empty struct |
| 13 | `interface{}` assignability | 1b | Low–Med | **Bug** — `any` works but `interface{}` doesn't |
| 14 | Comma-ok with `=` (map + type assert) | 1b | Low | **Bug** — only works with `:=` |

Total: 6 Tier-1 features + 6 Tier-1b bug fixes + 4 Tier-2 items. Tier-3 items are stretch goals.
